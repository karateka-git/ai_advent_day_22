# ТЗ: первый RAG-запрос для local-document-indexer

## 1. Цель текущего этапа

Реализовать в проекте первый полноценный RAG-сценарий и сравнение качества ответов модели:

- без retrieval;
- с retrieval по локальному индексу.

Главная учебная цель этапа:

- не просто получить работающий результат, а понять механику RAG на практике;
- увидеть, как вопрос превращается в embedding;
- понять, как находятся релевантные чанки;
- увидеть, как контекст влияет на итоговый ответ модели.

## 2. Главный учебный принцип

На этом этапе приоритет у прозрачности и понятности пайплайна, а не у максимальной производительности.

Поэтому первая реализация поиска должна быть такой:

- хранилище индекса: `SQLite`;
- retrieval: полный перебор embeddings;
- scoring: `cosine similarity`;
- генерация ответа: внешний LLM API по образцу `day_6`.

`FAISS` рассматривается как возможное продолжение после того, как базовый RAG станет понятен и стабилен.

## 3. Что должно получиться в итоге

Проект должен уметь:

- принимать пользовательский вопрос;
- отвечать без RAG;
- отвечать с RAG;
- показывать, какие чанки были найдены для RAG-ответа;
- поддерживать набор из 10 контрольных вопросов;
- формировать удобный материал для ручного сравнения качества.

Итоговый результат этапа:

- агент с двумя режимами: `plain` и `rag`;
- retrieval через `SQLite + cosine similarity`;
- 10 контрольных вопросов по текущей базе;
- ожидания по каждому вопросу;
- список ожидаемых источников;
- сравнение качества ответов модели без RAG и с RAG.

## 4. Границы текущего этапа

### Входит в текущий этап

- поиск по embeddings из локального индекса;
- расчёт cosine similarity в Kotlin;
- генерация ответа через тот же внешний LLM API, что использовался в `day_6`;
- два режима ответа: без контекста и с контекстом;
- вывод retrieval-результатов в CLI;
- набор из 10 контрольных вопросов;
- markdown-отчёт или аналогичный артефакт для сравнения качества.

### Не входит в обязательный минимум

- интеграция `FAISS`;
- hybrid search;
- reranking;
- web UI;
- автоматическая LLM-оценка качества;
- сложный orchestration pipeline для benchmark-ранов.

## 5. Архитектурное решение этапа

### 5.1. Разделение ролей компонентов

- `SQLite` остаётся основным локальным индексом.
- `Ollama` используется для embeddings.
- внешний LLM API из `day_6` используется для генерации ответа.
- retrieval выполняется в приложении через полный перебор embeddings.

### 5.2. Логика двух режимов

Режим `plain`:

- вопрос пользователя отправляется в LLM без retrieval-контекста.

Режим `rag`:

- вопрос пользователя превращается в embedding;
- из индекса выбираются релевантные чанки;
- чанки объединяются с вопросом в prompt;
- итоговый prompt отправляется в LLM;
- пользователю возвращается ответ и список использованных источников.

### 5.3. Почему сначала не FAISS

Причины:

- проще понять сам механизм retrieval;
- легче отлаживать релевантность и score;
- легче объяснить результат на защите или в отчёте;
- меньше инфраструктурной сложности на первом шаге.

## 7. Новые функциональные блоки

### 7.1. Search layer

Нужно добавить поисковый слой:

- `SearchEngine`
- `BruteForceSearchEngine`
- `CosineSimilarity`
- модели search results

Задача слоя:

- получить embedding запроса;
- загрузить embeddings чанков из индекса;
- посчитать cosine similarity;
- отсортировать результаты;
- вернуть `topK`.

### 7.2. Чтение индекса

Нужно расширить storage-слой возможностью читать из `SQLite`:

- чанки;
- embeddings;
- метаданные чанков;
- optionally фильтрацию по стратегии `fixed` или `structured`.

### 7.3. LLM generation layer

Нужно добавить отдельный клиент генерации ответа по аналогии с `day_6`.

Он должен:

- использовать `AGENT_ID` и `USER_TOKEN` из локального конфига;
- формировать chat/completion-запрос;
- поддерживать system prompt;
- возвращать текст ответа модели.

### 7.4. QA / RAG orchestration layer

Нужно добавить сервис, который умеет:

- отвечать без RAG;
- отвечать с RAG;
- собирать retrieval-контекст;
- формировать prompt;
- возвращать ответ вместе с retrieval-сводкой.

## 8. Требования к prompt-сценариям

### 8.1. Режим без RAG

Prompt должен содержать:

- системную инструкцию;
- вопрос пользователя.

### 8.2. Режим с RAG

Prompt должен содержать:

- системную инструкцию;
- блок контекста из найденных чанков;
- вопрос пользователя;
- явное требование опираться на контекст;
- явное правило честно говорить о нехватке данных.

### 8.3. Требование к ответу

Ответ модели в RAG-режиме желательно делать:

- кратким;
- опирающимся на факты из контекста;
- без выдумывания отсутствующих данных.

## 9. CLI-сценарий этапа

Нужно добавить новую пользовательскую команду, например:

```text
ask --query "..." --mode plain
ask --query "..." --mode rag --strategy structured --top 5
```

Минимально команда должна поддерживать:

- `--query`
- `--mode <plain|rag>`
- `--strategy <fixed|structured>`
- `--top <N>`

Режим `plain` может игнорировать `strategy`.

## 10. Retrieval-сценарий

### 10.1. Пайплайн

Пайплайн RAG-поиска:

1. Пользователь вводит вопрос.
2. Приложение получает embedding вопроса через `Ollama`.
3. Из `SQLite` читаются чанки и embeddings нужной стратегии.
4. Для каждого чанка считается cosine similarity.
5. Результаты сортируются по убыванию score.
6. Берётся `topK`.
7. Лучшие чанки попадают в prompt.
8. Prompt отправляется во внешний LLM.
9. Пользователь получает ответ и retrieval-сводку.

### 10.2. Что выводить пользователю

Для режима `rag` желательно показывать:

- итоговый ответ;
- `topK` найденных чанков;
- score каждого чанка;
- `filePath` или `title`;
- `section`;
- короткий preview текста.

## 11. Набор контрольных вопросов

### 11.1. Цель набора

Контрольные вопросы нужны не только для сдачи, но и для обучения:

- увидеть, где retrieval помогает;
- увидеть, где модель без контекста фантазирует;
- понять, как качество зависит от подбора чанков.

### 11.2. Требование к набору

Нужно подготовить 10 вопросов по текущему корпусу.

Для каждого вопроса нужно зафиксировать:

- `question`
- `expectation`
- `expectedSources`

Где:

- `expectation` — что обязательно или желательно должно появиться в ответе;
- `expectedSources` — из каких текстов должен браться ответ, если вопрос фактологический.

### 11.3. Формат хранения

Допустимые варианты:

- `data/control-questions.json`
- `data/control-questions.yaml`
- `docs/control-questions.md`

Предпочтительно использовать формат, который легко читать и вручную редактировать.

## 12. Сравнение качества

### 12.1. Что нужно сравнить

Для каждого контрольного вопроса нужно собрать:

- ответ без RAG;
- ответ с RAG;
- короткую оценку качества;
- наблюдение по использованным источникам.

### 12.2. Минимальный формат сравнения

Для каждого вопроса:

- вопрос;
- ожидание;
- ожидаемые источники;
- ответ `plain`;

### 12.3. Формат итогового артефакта

Предпочтительный вариант:

- `data/rag-evaluation.md`

В отчёте должно быть удобно глазами сравнить два режима.
